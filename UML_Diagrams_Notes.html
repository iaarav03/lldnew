📊 UML Diagrams for Low Level Design - Complete Guide

🎯 What is UML?

UML (Unified Modeling Language) is a standardized visual language for modeling software systems. Think of it as the "blueprint" for your code - just like architects use blueprints for buildings!

Why UML in Interviews?

  • 🎨 Visual Communication: Explain your design clearly
  • 🏗️ System Architecture: Show how components interact
  • 🤝 Relationships: Demonstrate understanding of object relationships
  • 💼 Industry Standard: Used everywhere in software development

💡 LLD Interview Success Tips

  • Start with Classes: Identify main entities first
  • Define Relationships: Is-A, Has-A, Uses relationships
  • Show Multiplicity: One-to-one, one-to-many, many-to-many
  • Use Proper Notation: Lines, arrows, symbols matter!
  • Explain Your Thinking: Walk through your design decisions

📚 UML Class Diagram Basics

🧱 What's in a UML Class?

A UML class is represented as a rectangle divided into three sections:

  1. Class Name (top section)
  2. Attributes/Fields (middle section)
  3. Methods/Operations (bottom section)

📐 Basic UML Class Structure

┌─────────────────────┐
│      ClassName      │  ← Class name (bold, centered)
├─────────────────────┤
│ - privateField      │  ← Attributes with visibility
│ + publicField       │
│ # protectedField    │
├─────────────────────┤
│ + publicMethod()    │  ← Methods with visibility
│ - privateMethod()   │
│ # protectedMethod() │
└─────────────────────┘

🔍 Visibility Symbols

  • + Public: Accessible from anywhere
  • - Private: Only within the same class
  • # Protected: Within class and subclasses
  • ~ Package: Within the same package/namespace

📝 Real Example: Car Class in UML

┌─────────────────────┐
│        Car          │
├─────────────────────┤
│ - brand: string     │
│ - model: string     │
│ - year: int         │
│ + speed: double     │
├─────────────────────┤
│ + startEngine()     │
│ + accelerate()      │
│ + brake()           │
│ - checkFuel()       │
└─────────────────────┘

Corresponding C++ Code:

class Car {
private:
    string brand;      // - private
    string model;      // - private  
    int year;          // - private
    bool checkFuel();  // - private method
    
public:
    double speed;      // + public
    
    void startEngine(); // + public method
    void accelerate();  // + public method
    void brake();       // + public method
};

🔗 UML Relationships - The Heart of Design

🤔 Why Relationships Matter?

In real software, classes don't exist in isolation. They interact, depend on, and contain each other. Understanding these relationships is crucial for:

  • 📈 Scalable Design: How your system grows
  • 🔧 Maintainability: How easy it is to modify
  • 🧪 Testability: How to unit test components
  • 🔄 Reusability: How to reuse components
Relationship UML Symbol Meaning C++ Example
Is-A (Inheritance) ──▷ (open triangle) "Child is a type of Parent" class Dog : public Animal
Has-A (Composition) ──♦ (filled diamond) "Owner owns and controls part" Car contains Engine (strong)
Uses (Aggregation) ──♢ (hollow diamond) "Uses but doesn't own" Student uses Library (weak)
Association ──── (simple line) "Knows about and uses" Person knows Address
Dependency - - -▷ (dashed arrow) "Temporarily uses" Method parameter/return

🧬 IS-A Relationship (Inheritance)

"A Dog IS-A Animal" - Inheritance Relationship

🤔 What is IS-A?

IS-A represents inheritance. It means the child class "is a specialized version" of the parent class. Think of it like family relationships:

  • 🐕 A Dog IS-A Animal
  • 🚗 A Car IS-A Vehicle
  • 👨‍🎓 A Student IS-A Person

📐 UML Notation for IS-A

        Animal
    ┌─────────────┐
    │   Animal    │
    ├─────────────┤
    │ + name      │
    │ + age       │
    ├─────────────┤
    │ + eat()     │
    │ + sleep()   │
    └─────────────┘
           △
           │ IS-A (inheritance)
           │ ──▷ (open triangle arrow)
           │
    ┌─────────────┐
    │     Dog     │
    ├─────────────┤
    │ + breed     │
    ├─────────────┤
    │ + bark()    │
    │ + wagTail() │
    └─────────────┘

💻 C++ Implementation of IS-A

// Base class (Parent)
class Animal {
protected:
    string name;
    int age;
    
public:
    Animal(const string& n, int a) : name(n), age(a) {}
    
    // Virtual functions for polymorphism
    virtual void eat() {
        cout << name << " is eating." << endl;
    }
    
    virtual void sleep() {
        cout << name << " is sleeping." << endl;
    }
    
    virtual void makeSound() = 0; // Pure virtual (abstract)
    
    virtual ~Animal() = default;
};

// Derived class (Child) - Dog IS-A Animal
class Dog : public Animal {
private:
    string breed;
    
public:
    Dog(const string& n, int a, const string& b) 
        : Animal(n, a), breed(b) {}
    
    // Override parent methods
    void makeSound() override {
        cout << name << " says: Woof! Woof!" << endl;
    }
    
    // Dog-specific methods
    void bark() {
        cout << name << " is barking loudly!" << endl;
    }
    
    void wagTail() {
        cout << name << " is wagging tail happily!" << endl;
    }
    
    string getBreed() const { return breed; }
};

// Another derived class
class Cat : public Animal {
public:
    Cat(const string& n, int a) : Animal(n, a) {}
    
    void makeSound() override {
        cout << name << " says: Meow! Meow!" << endl;
    }
    
    void purr() {
        cout << name << " is purring contentedly." << endl;
    }
};

// Usage demonstrating polymorphism
int main() {
    // Creating objects
    Dog myDog("Buddy", 3, "Golden Retriever");
    Cat myCat("Whiskers", 2);
    
    // Using IS-A relationship - Dog IS-A Animal
    Animal* animals[] = {&myDog, &myCat};
    
    cout << "=== Polymorphic behavior ===" << endl;
    for (Animal* animal : animals) {
        animal->eat();        // Inherited from Animal
        animal->makeSound();  // Polymorphic call
        animal->sleep();      // Inherited from Animal
        cout << endl;
    }
    
    cout << "=== Dog-specific behavior ===" << endl;
    myDog.bark();      // Dog-specific method
    myDog.wagTail();   // Dog-specific method
    cout << "Breed: " << myDog.getBreed() << endl;
    
    return 0;
}

📝 IS-A Interview Questions

  • Q: "When would you use inheritance?"
    A: When there's a clear "is-a" relationship and you want to share common behavior
  • Q: "What's the difference between public, private, and protected inheritance?"
    A: Controls access levels and IS-A relationship visibility
  • Q: "Why use virtual functions?"
    A: For runtime polymorphism - correct method called based on actual object type

🏠 HAS-A Relationship (Composition & Aggregation)

"A Car HAS-A Engine" - Ownership Relationship

🤔 What is HAS-A?

HAS-A represents ownership or containment. One class "has" or "contains" another class as a member. There are two types:

  • 💪 Composition (Strong HAS-A): Owner controls lifetime
  • 🤝 Aggregation (Weak HAS-A): Owner uses but doesn't control lifetime

💪 Composition - Strong HAS-A

🔧 Composition Characteristics

  • Ownership: Parent owns and controls child's lifecycle
  • Lifetime: Child cannot exist without parent
  • Example: Car HAS-A Engine, House HAS-A Room
  • UML Symbol: Filled diamond (♦)

📐 UML Composition Notation

    ┌─────────────┐       ♦ ──────── ┌─────────────┐
    │     Car     │       │          │   Engine    │
    ├─────────────┤       │          ├─────────────┤
    │ - brand     │       │          │ - horsepower│
    │ - model     │       │          │ - fuelType  │
    ├─────────────┤  Composition     ├─────────────┤
    │ + start()   │   (filled ♦)     │ + start()   │
    │ + stop()    │       │          │ + stop()    │
    └─────────────┘       │          └─────────────┘
                         
    "Car HAS-A Engine" (Car owns and controls Engine)

💻 C++ Implementation of Composition

// Child class - Engine
class Engine {
private:
    int horsepower;
    string fuelType;
    bool isRunning;
    
public:
    Engine(int hp, const string& fuel) 
        : horsepower(hp), fuelType(fuel), isRunning(false) {
        cout << "Engine created with " << hp << " HP (" << fuel << ")" << endl;
    }
    
    ~Engine() {
        cout << "Engine destroyed" << endl;
    }
    
    void start() {
        if (!isRunning) {
            isRunning = true;
            cout << "Engine started! Vroom vroom!" << endl;
        }
    }
    
    void stop() {
        if (isRunning) {
            isRunning = false;
            cout << "Engine stopped." << endl;
        }
    }
    
    bool getRunningStatus() const { return isRunning; }
    int getHorsepower() const { return horsepower; }
};

// Parent class - Car (HAS-A Engine)
class Car {
private:
    string brand;
    string model;
    Engine engine;  // Composition: Car HAS-A Engine
    
public:
    // Engine is created when Car is created (composition)
    Car(const string& b, const string& m, int hp, const string& fuel)
        : brand(b), model(m), engine(hp, fuel) {  // Engine constructed here
        cout << "Car created: " << brand << " " << model << endl;
    }
    
    // Engine is automatically destroyed when Car is destroyed
    ~Car() {
        cout << "Car destroyed: " << brand << " " << model << endl;
    }
    
    void startCar() {
        cout << "Starting " << brand << " " << model << "..." << endl;
        engine.start();  // Delegating to engine
        if (engine.getRunningStatus()) {
            cout << "Car is ready to drive!" << endl;
        }
    }
    
    void stopCar() {
        cout << "Stopping " << brand << " " << model << "..." << endl;
        engine.stop();   // Delegating to engine
        cout << "Car stopped." << endl;
    }
    
    void displayInfo() const {
        cout << "Car: " << brand << " " << model 
             << " (Engine: " << engine.getHorsepower() << " HP)" << endl;
    }
};

int main() {
    cout << "=== Composition Example ===" << endl;
    
    {  // Scope to demonstrate lifecycle
        Car myCar("Toyota", "Camry", 200, "Gasoline");
        myCar.displayInfo();
        
        myCar.startCar();
        cout << "Driving around..." << endl;
        myCar.stopCar();
        
        cout << "End of scope - Car will be destroyed..." << endl;
    } // Car destructor called -> Engine destructor called automatically
    
    cout << "After scope - both Car and Engine are destroyed" << endl;
    return 0;
}

🤝 Aggregation - Weak HAS-A

📚 Aggregation Characteristics

  • Usage: Parent uses child but doesn't own it
  • Lifetime: Child can exist independently of parent
  • Example: Student HAS-A Address, Team HAS-A Player
  • UML Symbol: Hollow diamond (♢)

📐 UML Aggregation Notation

    ┌─────────────┐       ♢ ──────── ┌─────────────┐
    │  University │       │          │   Student   │
    ├─────────────┤       │          ├─────────────┤
    │ - name      │       │          │ - name      │
    │ - location  │       │          │ - id        │
    ├─────────────┤   Aggregation    ├─────────────┤
    │ + enroll()  │   (hollow ♢)     │ + study()   │
    │ + graduate()│       │          │ + takeExam()│
    └─────────────┘       │          └─────────────┘
                         
    "University HAS-A Student" (University uses Student, but Student can exist independently)

💻 C++ Implementation of Aggregation

// Independent class - Student
class Student {
private:
    string name;
    int id;
    string major;
    
public:
    Student(const string& n, int studentId, const string& m) 
        : name(n), id(studentId), major(m) {
        cout << "Student created: " << name << " (ID: " << id << ")" << endl;
    }
    
    ~Student() {
        cout << "Student destroyed: " << name << endl;
    }
    
    void study() {
        cout << name << " is studying " << major << endl;
    }
    
    void takeExam() {
        cout << name << " is taking an exam" << endl;
    }
    
    string getName() const { return name; }
    int getId() const { return id; }
    string getMajor() const { return major; }
};

// Class that aggregates students
class University {
private:
    string name;
    string location;
    vector students;  // Aggregation: pointers to existing students
    
public:
    University(const string& n, const string& loc) 
        : name(n), location(loc) {
        cout << "University created: " << name << endl;
    }
    
    ~University() {
        cout << "University destroyed: " << name << endl;
        // Note: We don't delete students - they exist independently!
        cout << "Students continue to exist after university is destroyed" << endl;
    }
    
    // Enroll an existing student (aggregation)
    void enrollStudent(Student* student) {
        if (student) {
            students.push_back(student);
            cout << "Student " << student->getName() 
                 << " enrolled at " << name << endl;
        }
    }
    
    void removeStudent(int studentId) {
        auto it = remove_if(students.begin(), students.end(),
            [studentId](Student* s) { return s->getId() == studentId; });
        
        if (it != students.end()) {
            cout << "Student with ID " << studentId 
                 << " removed from " << name << endl;
            students.erase(it, students.end());
        }
    }
    
    void conductClass() {
        cout << "\n=== Class in session at " << name << " ===" << endl;
        for (Student* student : students) {
            student->study();
        }
    }
    
    void displayInfo() const {
        cout << "\n" << name << " (" << location << ")" << endl;
        cout << "Enrolled students: " << students.size() << endl;
        for (const Student* student : students) {
            cout << "  - " << student->getName() 
                 << " (ID: " << student->getId() 
                 << ", Major: " << student->getMajor() << ")" << endl;
        }
    }
};

int main() {
    cout << "=== Aggregation Example ===" << endl;
    
    // Create students independently
    Student* alice = new Student("Alice Johnson", 1001, "Computer Science");
    Student* bob = new Student("Bob Smith", 1002, "Mathematics");
    Student* charlie = new Student("Charlie Brown", 1003, "Physics");
    
    {  // University scope
        University mit("MIT", "Cambridge, MA");
        
        // Enroll existing students (aggregation)
        mit.enrollStudent(alice);
        mit.enrollStudent(bob);
        mit.enrollStudent(charlie);
        
        mit.displayInfo();
        mit.conductClass();
        
        // Remove a student (but student still exists)
        mit.removeStudent(1002);
        mit.displayInfo();
        
        cout << "\nUniversity scope ending..." << endl;
    } // University destroyed, but students still exist!
    
    cout << "\nAfter university destruction:" << endl;
    alice->study();    // Students still exist and work!
    charlie->study();
    
    // Manual cleanup (in real app, use smart pointers)
    delete alice;
    delete bob;
    delete charlie;
    
    return 0;
}

🔍 Composition vs Aggregation - Key Differences

Aspect Composition (♦) Aggregation (♢)
Relationship Strength Strong "owns" Weak "uses"
Lifetime Dependency Child dies with parent Child exists independently
Memory Management Parent creates/destroys child Child created elsewhere
C++ Implementation Member object or unique_ptr Pointer or reference
Real-world Example House has Rooms Library has Books
UML Symbol Filled diamond ♦ Hollow diamond ♢

🔗 Other Important Relationships

📞 Association - "Uses/Knows About"

Association is a general relationship where one class knows about and uses another, but there's no ownership.

Example: Person ──── Address (Person has an address, but address can exist without person)

💻 Association Example

class Address {
public:
    string street, city, zipCode;
    
    Address(const string& s, const string& c, const string& z)
        : street(s), city(c), zipCode(z) {}
    
    string getFullAddress() const {
        return street + ", " + city + " " + zipCode;
    }
};

class Person {
private:
    string name;
    Address* homeAddress;  // Association (pointer to existing address)
    
public:
    Person(const string& n) : name(n), homeAddress(nullptr) {}
    
    void setAddress(Address* addr) {
        homeAddress = addr;  // Associate with existing address
    }
    
    void displayInfo() const {
        cout << "Name: " << name << endl;
        if (homeAddress) {
            cout << "Address: " << homeAddress->getFullAddress() << endl;
        }
    }
};

⚡ Dependency - "Temporarily Uses"

Dependency is the weakest relationship - one class temporarily uses another (usually as parameter or return type).

UML Symbol: Dashed arrow - - -▷

💻 Dependency Example

class EmailService {
public:
    void sendEmail(const string& message) {
        cout << "Sending: " << message << endl;
    }
};

class NotificationManager {
public:
    // Dependency: temporarily uses EmailService
    void sendWelcomeNotification(EmailService& emailService, const string& userName) {
        string message = "Welcome " + userName + "!";
        emailService.sendEmail(message);  // Temporary usage
    }
    // EmailService not stored as member - just used temporarily
};

📊 Multiplicity in UML

🔢 What is Multiplicity?

Multiplicity shows how many instances of one class relate to instances of another class. It's written near the ends of relationship lines.

📐 Common Multiplicity Notations

1        Exactly one
0..1     Zero or one
1..*     One or more  
*        Zero or more (any number)
2..5     Between 2 and 5
5        Exactly 5

Example:
    Student ──────────── University
       *      studies at     1
    
"Many students study at one university"

🎓 Multiplicity Example: Student-Course System

class Course {
private:
    string courseName;
    string courseCode;
    vector enrolledStudents;  // * (many students)
    
public:
    Course(const string& name, const string& code) 
        : courseName(name), courseCode(code) {}
    
    void enrollStudent(Student* student) {
        enrolledStudents.push_back(student);
    }
    
    void displayEnrollment() const {
        cout << "Course: " << courseName << " (" << courseCode << ")" << endl;
        cout << "Enrolled students: " << enrolledStudents.size() << endl;
    }
};

class Student {
private:
    string name;
    int id;
    vector courses;  // * (many courses)
    
public:
    Student(const string& n, int studentId) : name(n), id(studentId) {}
    
    void enrollInCourse(Course* course) {
        courses.push_back(course);
        course->enrollStudent(this);  // Bidirectional relationship
    }
    
    void displayCourses() const {
        cout << name << " is enrolled in " << courses.size() << " courses" << endl;
    }
};

// Many-to-many relationship: Student * ──── * Course

💼 LLD Interview Scenarios

🎯 Common UML Questions in Interviews

  1. "Design a Library Management System"
    • Library HAS-A Books (aggregation)
    • Member borrows Books (association)
    • Book HAS-A Author (composition or association)
  2. "Design a Car Rental System"
    • Car IS-A Vehicle (inheritance)
    • RentalCompany HAS-A Cars (aggregation)
    • Customer rents Car (association)
  3. "Design a Social Media Platform"
    • User HAS-A Posts (composition)
    • User follows User (self-association)
    • Post HAS-A Comments (composition)

💻 Partial C++ Implementation

// Base class
class Vehicle {
protected:
    string id, brand;
public:
    virtual void start() = 0;
    virtual ~Vehicle() = default;
};

// IS-A relationship
class Car : public Vehicle {
private:
    int doors;
public:
    void start() override { cout << "Car engine started" << endl; }
    void openDoor() { cout << "Door opened" << endl; }
};

// HAS-A (Composition) - Rental owns vehicle reference
class Rental {
private:
    Vehicle* vehicle;  // Strong association
    Customer* customer;
    Date startDate, endDate;
public:
    Rental(Vehicle* v, Customer* c) : vehicle(v), customer(c) {}
    // Rental controls the rental relationship
};

// HAS-A (Aggregation) - Company uses vehicles
class RentalCompany {
private:
    vector fleet;  // Company has vehicles but doesn't own their lifecycle
    vector activeRentals;  // Company manages rentals (composition)
public:
    void addVehicle(Vehicle* v) { fleet.push_back(v); }
    Rental* createRental(Vehicle* v, Customer* c) {
        return new Rental(v, c);  // Company creates rental
    }
};

📝 UML Quick Reference Summary

🎯 Key Takeaways for LLD Interviews

  1. Start Simple: Identify main classes first
  2. Choose Right Relationships: IS-A for inheritance, HAS-A for containment
  3. Show Multiplicity: How many of each class relate to others
  4. Use Proper Notation: Symbols matter in technical discussions
  5. Explain Your Decisions: Why you chose each relationship type
When to Use Relationship C++ Pattern Example
Shared behavior, polymorphism IS-A (Inheritance) class Child : public Parent Dog IS-A Animal
Strong ownership, control lifecycle Composition Member object, unique_ptr Car HAS-A Engine
Uses but doesn't own Aggregation Pointer, reference, shared_ptr Team HAS-A Players
General relationship Association Reference, pointer Person knows Address
Temporary usage Dependency Parameter, return type Method uses Logger

🚀 Pro Tips for Success

  • Practice Drawing: Get comfortable with UML symbols
  • Think in Objects: Real-world entities become classes
  • Consider Lifecycle: Who creates/destroys objects?
  • Start High-Level: Main relationships first, details later
  • Validate Design: Can it handle the requirements?
